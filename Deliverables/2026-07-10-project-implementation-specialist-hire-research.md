# Hire Research: Project/Implementation Specialist

**Researcher:** Pax | **Briefed by:** Larry (step 1) via Nolan (SOP-001 step 2) | **Date:** 2026-07-10

## 1. What world-class looks like, day to day

The best-in-world version of this role sits between an **implementation manager** and a **delivery-side project manager**: implementation managers own the technical-to-business bridge (scope, cross-functional coordination, change facilitation), while delivery PMs own the supplier-side execution interface distinct from the client-side sponsor role ([Teal](https://www.tealhq.com/career-paths/implementation-manager); [APM](https://www.apm.org.uk/blog/project-management-skills-client-side-vs-delivery/)). Day to day this looks like: intake a new engagement into a structured scope/PRD; decompose it into work packages with owners and acceptance criteria; run a single risk/issue/change/decision register with a review cadence, not a write-once document; log every outbound client communication as sent-and-verified, not just drafted; and close engagements with a formal handover artifact, not a Slack message. Governance frameworks (PRINCE2, PMBOK) converge on the same shape: PRINCE2 explicitly separates "Controlling a Stage" (the PM authorizing work) from "Managing Product Delivery" (the team executing it) — work packages are formal authorization units, not to-do items ([PRINCE2](https://www.prince2.com/usa/blog/the-importance-of-governance-structures-in-project-delivery); [Wrike](https://www.wrike.com/project-management-guide/faq/what-are-the-differences-between-pmbok-and-prince2/)). This matches what Warwick already proved by hand in Fusion247 Brain's Bellrock project (numbered folder taxonomy, Work Package Catalogue, combined RAID-style register, Write-and-Verification Log, Support Handover Schema) — independent convergence, not coincidence.

## 2. Core competencies and anti-patterns

- **Scope decomposition into authorized work packages**, each with an owner and a done-state — not a task list (PRINCE2 doctrine).
- **Single-register discipline** for risks/issues/changes/decisions with named owners and a review cadence. The proven anti-pattern: "a RAID log nobody reviews is documentation theater, not management" ([RocketLane](https://www.rocketlane.com/blogs/raid-management)). Mediocre versions maintain the log for optics and never re-open it.
- **Provenance discipline**: raw sources (transcripts, client docs) are immutable; processed artifacts (PRDs, plans, extractions) carry source/date/confidence and are reconciled back into the live register, never silently overwritten.
- **Communication verification**, not just communication. Mediocre delivery roles treat "sent" as "done"; world-class roles log and check outbound writes before treating them as final.
- **Anti-pattern to explicitly avoid**: treating dashboards/boards/bots as sources of truth. They are views onto the register, never the register itself — a rule the user's own prior system already enforced and one this specialist should inherit verbatim.
- **Anti-pattern**: acting as an autonomous executor. World-class delivery PMs receive structured work packages and route judgment calls (scope, risk acceptance, closure) to the human — never silently expand scope or auto-close risk items.

## 3. Deliverables — adequate vs world-class

Adequate: a scope doc, a task list, occasional status updates. World-class: a **Project PRD** (intent, not just tasks), an **Implementation Plan**, a **Work Package Catalogue + Library** (each WP independently referenceable), a **combined Risk/Issue/Change/Decision register** with dated reconciliation entries when new source material lands, a **Write-and-Verification Log** for every outbound client artifact, and a **Support Handover** document at closure — a structured schema, not a farewell email. The differentiator is traceability: any current-state claim should be answerable by pointing to the register or the plan, never "I think we agreed that."
