## 1. Verified current state

**GL-006 / `Client Delivery/` today.** Three entity types exist: Engagement, Work Package, Register Item. Register Item's `kind` field is a closed enum — confirmed by direct read of the schema block (GL-006 §"Entity schemas"): `risk | issue | change | decision`. No Action, Configuration Change, Milestone, Meeting, Question, Source, or Uncertainty entity type exists. Each of these is either named in GL-006's own "Future extension candidates" section (Meeting Metadata, Deliverables, Requirements, Dependencies, Action) or flagged in "Known gaps" (verification metadata, `owner_side`) — GL-006 already documents its own incompleteness here; this is not a new discovery.

Register Item already carries `source_ref`, `evidence_type`, `confidence`, and `reread_flag` — a genuine evidence-provenance model, added in GL-006 v1.1, independent of and prior to this evaluation. This matters for §3 below: several of the live BRK-001 sheet columns this evaluation is asked to reconcile are not gaps at all — they are duplicate expressions of fields GL-006 already has.

**SQLite mirror.** Confirmed by direct read of `SOP-002-convert-mypka-to-sqlite.md`: its folder map, table list (`people`, `organizations`, `topics`, `projects`, `key_elements`, `goals`, `habits`, `journal`, `documents`, `journal_media`, `content_index`), and every SQL snippet are scoped entirely to `PKM/`. `Client Delivery/` does not appear anywhere in the file. GL-006 §"Does this feed the SQLite mirror?" says the same thing in its own words: "Not yet, and this is a real open decision, not a technicality."

**Cockpit.** Confirmed by direct grep of `Expansions/mypka-cockpit/sqlite-extension/DATA-CONTRACT.md`, `docs/db-contract.md`, `server/queries.js`, and `server/db.js`: zero matches for "client delivery," "engagement," "register_item," or "warden." Nothing under `Client Delivery/` feeds any Cockpit view today.

**Warden's actual operational workflow.** Confirmed by direct read of Warden's contract and `SOP-010`–`014`: the five meeting-intelligence SOPs already implement much of what the Sheets doc and IDEA-003 argue for — a single full read producing durable structured output (SOP-010), a reusable Evidence Pack instead of repeated transcript rereads, targeted-reread discipline tied to named triggers (not "reread for general context"), and a source-tier precedence doctrine (contract > baseline > evidence > structured knowledge > generated output) that already governs conflict resolution. This is a genuine point of convergence, not a competing design imported wholesale — see §3 and §4.

**Live BRK-001 usage (from the sanitized abstraction, not the raw workbook).** The BRK-001 engagement is running four real Google Sheets workbooks today: the live register (`Decisions`, `Actions`, `Risks_Issues`, `Config_Changes`, `Open_Questions`, `Milestones`, `Entities`, `Sources`, `Uncertainty`), a Write-and-Verification Log workbook, a branch variant carrying `work_package_id`/`work_package_name` columns on every RAID sheet, and a template-level Work Package Machine Index workbook. Row counts are small (single/low-double-digit per sheet) — this is early, real usage, not a stress test of either architecture.

## 2. Confirmed operational gaps

Distinguishing gaps that are genuinely unsupported from gaps that are merely undocumented:

1. **Actions have no entity type.** SOP-010 §7 and SOP-011 §4 both explicitly document the workaround (map an action onto a `decision` Register Item's `owner`/`target_resolution_date`, or a Work Package's `target_date`) and both explicitly flag it as a "watch, don't fix yet" gap. This is a real, named, currently-tolerated gap — not a surprise.
2. **No dedicated queryable layer for `Client Delivery/` at all.** Grep-based lookup only. For a single small engagement (today's BRK-001 row counts) this is not yet a performance problem; it is a *retrieval-ergonomics* problem — "give me all overdue customer actions" or "what changed since the last review" requires a human or an LLM to open and reason over multiple markdown files rather than running one filtered query.
3. **Write-and-verification enforcement is policy-only.** GL-006 §"Known gaps" #1 already says this in its own words: no `created_by`/`review_status`/`reviewed_by`/`reviewed_date` field exists on Register Item, so Warden's Critical rule 8 ("writer never self-verifies") has no machine-checkable trail. The live BRK-001 Write-and-Verification Log workbook (`write_manifest` + `verification_log` sheets, with `verified_by`/`verified_at`/`verification_status` columns) already implements almost exactly this. This is the single sharpest, most concrete gap this evaluation surfaces — a working reference implementation already exists in the live engagement, in a different medium.
4. **`owner_side` (internal/client/third-party) is only partially derivable.** GL-006 §"Known gaps" #2 already documents this; SOP-011 §4 already states the best-effort workaround and its limits. Not new.
5. **Milestones, Open Questions, and Configuration Changes have no discrete home**, and their content is currently absorbed into Register Item `kind: change`/`decision` or Work Package fields with some loss of shape (a milestone is a date-and-commitment fact, not naturally a risk/issue/change/decision; a configuration change's `system_environment`/`made_by` shape doesn't map cleanly onto Register Item's generic fields). This is a real fit gap, not merely an absent nice-to-have — see §3.
6. **No structured Sources/Entities register.** `Sources (Immutable)/INDEX.md` covers documents; there is no equivalent structured register for *people/systems/orgs mentioned in evidence* (the BRK-001 `Entities` sheet) with disambiguation/alias tracking. This is a genuinely new capability, not present even conceptually in GL-006 today.
7. **No dedicated Uncertainty register.** GL-006's `confidence`/`reread_flag` fields on Register Item cover uncertainty *about a specific extracted item*. The BRK-001 `Uncertainty` sheet (`u_id`, `item`, `why_uncertain`, `resolve_by`, `related_ids`) tracks something more general — an open question about the state of the world that may not map to any single register item yet. Adjacent to Open_Questions; possibly the same underlying need expressed twice in the live sheets themselves (a live-system hygiene observation, not a schema requirement to copy uncritically).

**What is not a gap, despite reading that way in the Foundry material (per Larry's account) and the Sheets doc:** stable IDs (GL-006's engagement-prefixed compound slugs already are this), narrative-outside-the-record (GL-006's frontmatter/body split already is this), evidence anchors (`source_ref`/`evidence_type`/`confidence`/`reread_flag` already are this, and predate this evaluation), and query-recipe-shaped skills (SOP-011/012/013/014 already are this, in markdown-native form). The Sheets doc's own argument and GL-006's existing design converge on more of these principles than either side seems to have recognized before this evaluation.

| **Risks_Issues** | retain | GL-006's `kind: risk`/`kind: issue` already covers this directly, including `severity`, `cause`≈body prose, `mitigation`≈body prose. |

## 4. Architecture options

Assessed against the same evidence base — GL-006's actual current shape, Warden's actual SOPs, and the live BRK-001 usage — not against which option "sounds more modern."

**(a) Current structured Markdown, no dedicated query mirror (status quo).**
- *For:* Zero additional build or maintenance surface. SOP-010–014 already deliver most of the Sheets doc's stated goal ("give me the ten things that matter tomorrow") without a spreadsheet, via targeted derivation from frontmatter + body. Markdown stays the single, git-diffable, human-and-LLM-readable source of truth — no second system to keep in sync.
- *Against:* At BRK-001's current scale (dozens of items per sheet) this is fine; it will not obviously survive the "600 actions, 250 risks, 200 decisions" scale the Sheets doc itself uses as its working example, because there is no filtered-query primitive — every retrieval is "read the relevant files and reason," not "run a WHERE clause." No queryable cross-engagement view exists (e.g. "every critical risk across all active engagements") without opening every engagement folder.
- *Verdict:* Sound for exactly one small-to-medium engagement. Not proven at the scale IDEA-003 (per Larry's account) and the Sheets doc both anticipate, but that scale doesn't exist yet either — BRK-001's real row counts today are small.

**(b) Structured Markdown with a generated SQLite mirror (extend SOP-002's own pattern to `Client Delivery/`).**
- *For:* Directly answers the retrieval-ergonomics gap (§2.2) with a filtered-query primitive, using an architecture the team already owns, already trusts, and already knows how to regenerate idempotently (SOP-002). Markdown stays canonical exactly as GL-006 already requires; SQLite stays derived and disposable, consistent with Silas's whole operating philosophy. Naturally extends to cross-engagement queries the moment more than one engagement exists.
- *Against:* Someone has to actually write and maintain the mapping (folder map → tables → validation queries), and keep it in sync with GL-006 every time GL-006 changes — a real, ongoing maintenance cost SOP-002 doesn't currently carry for `Client Delivery/` at all. Does not by itself solve the write-and-verification enforcement gap (§2.3) — a SQLite mirror is read-side; the missing fields (`created_by`/`review_status`) still have to be added to GL-006 first, independent of which query layer reads them.
- *Phone-access relevance:* The original IDEA-003 exploration document states the Sheets approach "was originally conceived as a phone-friendly temporary layer while building Fusion247 Brain operations and project management on-device with ChatGPT" and explicitly asks whether that constraint is still relevant. This matters here: the Cockpit expansion (the natural consumer of this SQLite mirror) already has a documented LAN/phone-access mode (`Expansions/mypka-cockpit/SECURITY.md`: `COCKPIT_BIND_LAN=1`, described as binding `0.0.0.0` for the user's own phone/tablet) and a genuinely mobile-responsive UI (`web/src/components/Sidebar.tsx` has an explicit mobile off-canvas drawer, not a desktop layout merely squeezed down). This is real and verifiable, not aspirational — it's evidence this path doesn't sacrifice phone access to get a query layer, once Client Delivery is actually wired into it. It is *not* proof the Cockpit's phone story is already adequate for this specific operational-register use case — that generic capability existing is not the same as it being tested against Client Delivery's actual usage pattern; see §7.
- *Verdict:* The natural next step *given the current Markdown-first direction GL-006 already committed to* — it extends rather than replaces the existing design. Directly deprioritized in the Foundry's implied sequencing (per Larry's account, WP2 out of six, after a schema-reconciliation pass) — this evaluation does not accept that sequencing as given either; see §6.

**(c) Google Sheets as an interim operational layer (the Sheets doc's own proposal, and IDEA-003's origin per Larry's account).**
- *For:* Already running, already proven at BRK-001. The Sheets doc's core argument — atomic rows, stable IDs, selective retrieval instead of full-tab reads, narrative kept outside the sheet, a CSV exit ramp — is genuinely sound engineering, and large parts of it are *already true of GL-006* independently (see §2, "what is not a gap"). A human PM can eyeball a filtered view directly, no LLM required — a real advantage Markdown-plus-SQLite doesn't have.
- *Against:* This is a second, structurally different source-of-truth mechanism for the exact same category of fact GL-006 already governs — directly against Warden's own philosophy #5 ("views are not the source... the register itself is canonical") and against this whole myPKA's markdown-canonical operating principle (Silas's own core philosophy #1). Introducing Sheets as *the* register, rather than as a generated view of the Markdown register, means two canonical stores for register data existing simultaneously, which is exactly the "parallel register branch got forked" failure Warden's Critical rule 8 names as a real, previously-lived incident. Any conflict between a Sheets row and a Markdown Register Item would need its own reconciliation doctrine that doesn't exist yet.
- *Verdict:* Valuable as **existing evidence of what real usage needs**, not as a replacement source of truth. Adopting it as the primary register would be a genuine regression against this myPKA's own SSOT doctrine, not merely "a different valid choice." Its principles (atomic records, stable IDs, selective retrieval, narrative elsewhere, CSV exit ramp) are worth retaining regardless of which storage medium wins — and mostly already are retained, in GL-006's own current design.

**(d) Hybrid — Markdown stays canonical; Sheets (or a generated CSV/Sheets *view*) is a read-only, regenerated export for human PM use, same relationship SQLite already has to Markdown.**
- *For:* Gets the human-scannable, filterable table experience the Sheets doc argues for, without creating a second canonical store — Sheets becomes a *rendering*, exactly like the Cockpit is a rendering of PKM data today, not a competing source. Markdown stays the single place a schema change or a conflict-resolution rule needs to be taught. BRK-001's live Sheets usage doesn't need to be retired or migrated at all under this option — it can keep running as-is while a generated view is evaluated separately, deferring any decision about the *live* BRK-001 Sheet (which this evaluation is explicitly barred from touching).
- *Against:* Building a Markdown→Sheets export is genuinely new work with no existing SOP to extend (SOP-002 targets SQLite, not Sheets/CSV) — a real build cost, not a free option. Two derived layers (SQLite for LLM/Cockpit queries, Sheets/CSV for human PM scanning) is more moving parts than picking one.
- *Phone-access relevance:* Same consideration as (b) applies here, and arguably more directly — if the Cockpit's LAN/mobile support (§4b) turns out to already deliver the phone-friendly access the original Sheets proposal was chasing, a second regenerated Sheets/CSV export for the same purpose becomes redundant scope rather than a distinct benefit. Worth testing (b)'s phone story first before committing to this option's extra build cost for the same need.
- *Verdict:* Plausible, but strictly less proven and less immediately useful than (b) given no live pain point about human-scannability has actually been reported for `Client Delivery/` yet (the operational gaps in §2 are LLM-retrieval-ergonomics and write-verification gaps, not "the PM can't read a Markdown file" gaps). Worth naming as a real fourth option rather than a straw man, but not the one this evaluation would spend the first proof on.

## 5. Recommendation

**Preferred direction: stay Markdown-canonical (GL-006 unchanged in its fundamental shape), close the two concretely-evidenced gaps first, and treat a SQLite mirror extension as the natural, but not yet urgent, next infrastructure step — never adopt Sheets as a second source of truth.**

Evidence for this, weighed plainly:

- GL-006's current design is **more aligned with the Sheets doc's own stated principles than either side recognized** — stable prefixed IDs, evidence anchors, narrative-outside-the-record, and query-recipe-shaped skills are already real, not proposed. The genuine architecture question is not "Markdown vs. Sheets" (the Sheets doc's own framing) — it's "does Markdown need a queryable mirror yet," which is a narrower, already-precedented question SOP-002 already answers for `PKM/`.
- The single most concrete, most corroborated gap this evaluation found is **not** a missing entity type or a missing query layer — it is the **write-and-verification enforcement gap** GL-006 already named as a known gap, for which the live BRK-001 Write-and-Verification Log workbook is a working reference implementation of the mechanics. But naming a well-evidenced candidate is not the same as GL-006 having already decided to implement it — GL-006 records the gap and names the candidate fields, it does not contain an approved decision to add them. This should be treated as a **schema experiment / testable hypothesis**, proposed to run alongside or after the existing synthetic-engagement validation gate `tsk-2026-07-11-002` already tracks as a confirmed merge blocker — not as closing out a decision GL-006 has already made. The synthetic engagement can prove the *mechanics* work (fields populate, regeneration doesn't break); actually deciding to keep the fields still needs real second-engagement evidence, per GL-006's own stated discipline.
- The Actions/Milestones/Open-Questions/Config-Changes/Write-and-Verification-Log gaps are all real but are each evidenced by **exactly one engagement** — precisely the condition GL-006's own v1.1–v1.4 version history has repeatedly and explicitly declined to schema against ("don't over-fit the schema to one engagement... needs real data from a second engagement"). This evaluation should not reverse that discipline on IDEA-003's say-so (per Larry's account) any more than it should on the Sheets doc's — and applies it evenly across all five, rather than exempting the write-and-verification gap because its workaround happens to look more finished.
- Adopting Sheets as the operational layer would directly contradict two things this myPKA has already decided, independently of this evaluation: Warden's philosophy #5 ("views are not the source") and Silas's own philosophy #1 ("Markdown is canonical... every other shape is derived"). Overturning both would need much stronger justification than "a real client engagement is already using Sheets for now" — that fact supports extracting *principles* from the live usage, not adopting its storage medium.

## 6. Smallest useful next proof

**Not the Foundry's WP1 (schema reconciliation) by default — chosen independently, on the evidence above.**

**Recommended next proof: a single, scoped Silas schema-evolution proposal (not yet authorized, not yet built), framed as a schema experiment / testable hypothesis, to add candidate machine-checkable write-and-verification metadata to Register Item** — the specific fields GL-006 §"Known gaps" #1 already names (`created_by`, `review_status`, `reviewed_by`, `reviewed_date`), or a lightweight standalone Write-and-Verification Log entity type per GL-006's own extension path (§"How to extend this Guideline," path 2) — proposed to run *alongside or after* the existing synthetic/redacted worked-example `Client Delivery/` engagement that `tsk-2026-07-11-002` already tracks as a confirmed merge blocker, not against live BRK-001 data, and not ahead of that blocker (see §8).

This is deliberately **not** framed as closing out a decision GL-006 has already made. GL-006 names the gap and the candidate fields; it does not contain an approved decision to implement them. The synthetic engagement, once it exists, can test the *mechanics* — do the fields populate, does regeneration still work, does the shape hold up structurally. It cannot by itself satisfy GL-006's own "needs a second real engagement" evidence bar for actually deciding to keep the fields — that still requires real second-engagement usage, the same discipline this evaluation applies to Actions, Milestones, and Open Questions.

Why this one, not a broader schema pass and not a SQLite mirror first:

- It is the **only** gap in this evaluation with a working reference implementation of the *mechanics* already running in real use (the BRK-001 Write-and-Verification Log workbook), rather than resting on one engagement's still-forming usage pattern (Actions, Milestones, Open Questions) — that makes it the best candidate for testing mechanics now, not a candidate for skipping the second-engagement evidence bar.
- It is a **named, dated, already-documented** candidate in GL-006 itself (v1.2, "Known gaps" #1) — this is testing a well-scoped, pre-identified hypothesis, not inventing a new one, but it is still a hypothesis to be tested, not a decision to be finished.
- It is schema-only: no SQLite, no Cockpit, no retrieval skill, no BRK-001 touch — squarely inside what a next step can actually be authorized to do without re-litigating the SQLite-mirror or Sheets-vs-Markdown question at all.
- It directly strengthens the thing this repository's own migration-closure audit already flagged as unresolved (Warden's Critical rule 8 has no machine-checkable trail) — so this proof pays down existing, tracked technical debt rather than adding new scope.

Explicitly **not** recommended as the next proof, and why:
- **A SQLite mirror extension** (§4b) is the right *second* step, but only once `Client Delivery/` has more than one real engagement's worth of data to mirror — mirroring one engagement's dozen-item register proves little about the query-ergonomics problem it exists to solve.
