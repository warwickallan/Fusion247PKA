## What this is

The user built a working business/client-delivery governance system by hand in a prior tool ("Fusion247 Brain," ChatGPT-orchestrated, Google Drive-based) — proven live on a real client project (Bellrock / BRK-NPL-001, NPL Concerto implementation). This task **is** that record; update it every session rather than starting a new tracking doc.

2. **2026-07-10 — Name: Warden, role: Delivery Manager.** Chosen from Pax's candidates (Atlas, Crane, Warden, Keel). No roster collision. Approved by user.
3. **2026-07-10 — New top-level root `Client Delivery/`, sibling to `PKM/` and `Team Knowledge/`, not nested under `PKM/`.** User explicitly wants business projects structurally separate ("separate dashboard to Tom's personal life stuff"). Validated after the fact: the user's own Fusion247 Brain PRD §8.9 ("Split Brains / Domains") had already anticipated this separation independently.
4. **2026-07-10 — `Client Delivery/` entities get a sibling Guideline (`GL-006`), not an extension of `GL-002`.** GL-002's own extension path assumes a new `PKM/` folder; `Client Delivery/` is deliberately outside PKM. GL-006 inherits GL-002's mechanical rules (snake_case, ISO dates, slug-FK convention) by reference, never duplicates them. Approved by user.
5. **2026-07-10 — Three entity types for v1: Engagement, Work Package, Register Item.** Register Item combines risk/issue/change/decision via a `kind` field rather than four separate entity types — matches Warden's own "single combined register" language literally. Engagement note *is* the Project PRD. Write-and-Verification Log and Support Handover explicitly deferred (no schema yet). Whether `Client Delivery/` ever joins a SQLite mirror is deferred, not defaulted into `SOP-002`.
6. **2026-07-10 — Nothing gets copied verbatim from Fusion247 Brain.** Every contract/Guideline/SOP re-derives the underlying principle in myPKA's own voice and layering (agent = identity/boundaries only; procedure → SOP; shared rule → Guideline; output shape → Template; recurring multi-agent flow → Workstream). This mirrors a refactor the user's own Fusion247 Brain had independently proposed (`F247.proposal.agent-skill-boundary-refactor`, 2026-07-09) before this task started.
7. **2026-07-10 — Reconciliation pass triggered mid-build.** User surfaced two more source documents from the live Fusion247 Brain: an overnight 12-phase build proposal from "Fusion" (ChatGPT) re-architecting the Bellrock project folder taxonomy, and a QA verdict on it from "Fable" (a manually-run Claude instance acting as external QA — not a myPKA agent). Fable's verdict: the proposal's principles are sound (source-tier precedence, evidence/confidence/reread-flag discipline, one-evidence-pack-not-three-VTT-rereads) but it drops the `Project Control` folder without explicitly re-homing six artifacts (business case, project brief, comms plan, benefits log, stakeholder register, document register) that a prior cleanup had fought to preserve. Fable also caught **live governance drift already happening in Fusion247 Brain itself**: a duplicated/parallel register branch, a write logged only in a project-local manifest (not the Brain's own Session Log) sitting at `pending_verification`, and an apparently-deleted (not archived) README. This is Fusion's/the user's live Drive state, out of reach and out of scope for myPKA to fix directly — but it is the negative worked example for a hard rule going into Warden's contract: never fork a register, verification never comes from the writer, every write logs centrally.
8. **2026-07-10 — Sequencing: schema enrichment now, four-skill SOPs after a second engagement exists.** Per the user's own Fusion247 Brain PRD §11.2/§15.8 ("Markdown folders before databases," "avoid sharpening the axe forever") — land the source-tier doctrine + evidence/confidence fields + the six-artifact homes into GL-006 now; build the four meeting-intelligence skills (Meeting Prep, Configuration Guide, Meeting Summary, Consultant Summary) as Warden SOPs once there's a second real workload to prove the schema isn't over-fit to Bellrock. **User overrode this on 2026-07-10: the four skills are not overkill (user saw the actual risk-register output quality from the Fusion247 Brain pilot) — build them now, not deferred.** Superseding decision — see Updates.
9. **2026-07-10 — Stakeholder register realized as a list-of-objects frontmatter field (`linked_stakeholders`: person/role/influence/cadence), not a new entity type, and not a flat parallel-list shape.** GL-002 rule 4's foreign-key convention assumes flat slug lists; a stakeholder register needs per-person attributes (role, influence, cadence) that have no meaning outside their own entry. Nesting them under each `linked_stakeholders` entry avoids index-matching fragility (parallel `linked_stakeholder_roles[0]` arrays). Scoped to GL-006 only — not proposed as a GL-002-wide typing change. Flagged for Larry/the user to double-check: this is the one deliberate departure from the flat-slug-list convention in this schema.
10. **2026-07-10 — Document register resolved as `Sources (Immutable)/` plus a new per-engagement `INDEX.md`, not a new entity type.** A table (title, capture date, source tier, description, wikilink) covers the register need without frontmatter or a SQLite table. Flagged as a judgment call worth double-checking: if `Client Delivery/` later needs to query documents structurally across engagements, this is the first thing to promote to a real entity type (see GL-006 §"Future extension candidates").
11. **2026-07-10 — Built the four meeting-intelligence skills now, per decision 8's override, as `SOP-010` through `SOP-014` (Warden-owned, house SOP format, re-derived in myPKA's own voice — nothing copied verbatim from the Fusion247 Brain build doc).** `SOP-010-warden-extract-source-to-evidence-pack` is the foundational skill (one full read → Register Items with correctly-populated `evidence_type`/`confidence`/`reread_flag`, plus a reusable Evidence Pack); `SOP-011-warden-meeting-prep`, `SOP-012-warden-configuration-guide`, `SOP-013-warden-meeting-summary`, and `SOP-014-warden-consultant-summary` all read the Evidence Pack first and reopen the source only per GL-006's own `reread_flag` trigger list — no new trigger categories invented. Each of SOP-011–014 carries its own QA/anti-embellishment checklist tailored to that skill's actual failure mode (Meeting Prep: ranking discipline and reread restraint; Configuration Guide: no invented Environment/Dependencies, every step gets a validation check; Meeting Summary: "agreed"/"confirmed" gated on `evidence_type: agreed-decision`; Consultant Summary: fact vs. interpretation visibly separated, tier-1 cross-check, internal-only unless the user explicitly approves external circulation). Two judgment calls made along the way, flagged for double-checking:
    - **Evidence Pack home/tier.** Classified as **tier 4 (structured project knowledge)** under GL-006's precedence doctrine — it's a processed distillation of a source, not raw evidence itself — but housed physically at `Sources (Immutable)/Evidence Packs/<source-slug>-evidence-pack.md` for 1:1 discoverability with the source it describes, on the same precedent GL-006 already set by tolerating the (also processed) document-register `INDEX.md` inside that same folder. Kept in a named subfolder specifically so raw captures and processed packs stay visually distinct. Not treated as immutable itself — a reread pass appends a dated `## Reread log` entry rather than silently rewriting the prior extraction, mirroring a Register Item's `## Reconciliation log`.
    - **Configuration Guide save location.** Placed at `Work Packages/<engagement-slug>-configuration-guide.md` — same folder, same plain-markdown-rollup shape as the existing Work Package Catalogue — rather than a new folder or embedding into individual Work Package files, on the reasoning that configuration items already have canonical homes (Register Items own state, Work Packages own scope) and the guide is a derived execution view, not a new thing to be foreign-keyed into.
    Also surfaced, not blocking: myPKA's Register Item schema has no discrete `action` kind (risk/issue/change/decision only) — "overdue actions" in Meeting Prep are read off open Register Items' `target_resolution_date` and Work Packages' `target_date` instead. Flagged for Silas via Larry as a minor schema-fit gap to watch once a second real engagement exists, not urgent enough to act on now. Updated: `Team Knowledge/SOPs/INDEX.md` (SOP-010–014 rows, reserved range bumped to SOP-015+), `Team/Warden - Delivery Manager/AGENTS.md` (routing table, §Method step 3, new §Meeting-intelligence skills, References).
12. **2026-07-10 — Activated Codex CLI as a second host for the whole team, purely for the user's token-budget reasons (running the same myPKA team from Codex CLI as well as Claude Code) — not an architectural change.** Per [[GL-005-llm-agnostic-portable-core]] and SOP-001 Step 5's host-shim pattern, this is exactly the additive, per-host adapter work the portable-core boundary exists to make cheap: the wiki contracts (`Team/<Name> - <Role>/AGENTS.md`) and every Guideline/SOP/Workstream in the portable core are unchanged; only a new adapter directory (`.codex/agents/`) was added. Two sub-decisions:
    - **Corrected SOP-001's Codex CLI host-table row**, which had gone stale (it said `.codex/agents/<slug>.md (if supported by active version) — otherwise add a one-line specialist roster note in AGENTS.md.codex`). Per Pax's 2026 research, Codex CLI supports real per-repo named subagents today via project-scoped TOML files at `.codex/agents/<slug>.toml` (only loads in "trusted" projects), each requiring `name`, `description`, `developer_instructions`, and optionally `model`, sandbox policy, and MCP servers — genuine parallel dispatch, same category of capability as Claude Code's `.claude/agents/*.md`. The stale "otherwise… roster note" fallback is removed; the table now points at the correct TOML format so the *next* hire doesn't inherit the drift.
    - **Backfilled `.codex/agents/<slug>.toml` for all 12 currently-hired specialists** (Nolan, Pax, Penn, Mack, Silas, Felix, Vex, Vera, Iris, Charta, Pixel, Warden — Larry has no subagent shim in either host since he's the orchestrating session itself, not a dispatched subagent) by translating each specialist's existing `.claude/agents/<slug>.md` shim content into TOML (never re-derived from the canonical `AGENTS.md` contracts, to avoid drift between the two host shims for the same specialist). No `model`, sandbox-policy, or MCP-server fields were set for any specialist — none of the existing Claude Code shims pin a model, so there was no equivalent intent to carry over, and inventing one would violate the "never invent ad-hoc fields" discipline already established for this build (GL-002, GL-006). `Team/agent-index.md` was checked and does not track per-host shim coverage, so it was left as-is rather than speculatively adding a column.
