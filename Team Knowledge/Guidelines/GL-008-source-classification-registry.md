# GL-008 - Source Classification Registry

> **This Guideline is a general rule every agent reads on every relevant action.** Cairn reads this file at classification time on every intake (§Method step 2 of [[Team/Cairn - Knowledge Intake Specialist/AGENTS]]). Any other specialist who occasionally files an already-acquired external source into `PKM/` reads it too, rather than improvising a category. SOPs and Workstreams `[[wikilink]]` here rather than restating the vocabulary.

This is the source of truth for the small, governed set of **source-type categories** used to classify already-acquired external material before it is filed into a `PKM/` entity note. It answers one question, and only one: *what kind of source is this, and which of GL-002's eight entity types does it typically map to.*

## Why this exists

Cairn's own contract names the gap directly: "the vocabulary itself is not Cairn's to invent solo" ([[Team/Cairn - Knowledge Intake Specialist/AGENTS]] §Core philosophy 1). Without a governed list, every session reinvents its own bucket for "another YouTube transcript" or "another PDF," and the categories drift note by note until search and pattern-recognition both degrade — the exact anti-pattern Pax's hire research names as the most commonly cited cause of PKM rot: ungoverned, per-item classification ([[2026-07-11-knowledge-intake-synthesis-hire-research]] §1-2, citing Enterprise Knowledge and taxonomystrategies.com on taxonomy governance, and Forte Labs / dsebastien.net on tagging anti-patterns).

The user's own prior system (Fusion247 Brain) had already attempted this once: `F247.template.source-classification-registry.docx`, read directly for structural precedent while drafting this file. That template built a five-dimension classification matrix (source form, content domain, context/ownership, intended output, sensitivity) and a template-selection table, but was never more than an AI-authored draft awaiting the user's review, and it classified toward *templates* in a different system, not toward myPKA's eight existing entity types. This Guideline re-derives the underlying principle — inspect the source, classify it against a small fixed vocabulary, then map to the correct destination — in myPKA's own shape: a stable category list mapped onto GL-002's existing entity schemas, not a new template system and not a second classification axis. Professional taxonomy practice holds top-level categories to roughly 4-7, grown only from proven usage under an explicit steward, not per-item ([[2026-07-11-knowledge-intake-synthesis-hire-research]] §1). This Guideline is that governed list.

## What this is explicitly NOT

**This registry does not create a new entity type and does not add any new frontmatter field.** Cairn (and anyone else filing external material) still files exclusively into the eight existing `PKM/` entity types — Person, Organization, Project, Goal, Habit, Topic, Key Element, Document — exactly as [[GL-002-frontmatter-conventions]] defines them, unchanged. The category a source is classified under here never appears as a frontmatter key on the destination note. It is filing *guidance*, stated once in the note's body as the one-line classification decision Cairn's Method step 2 already requires, not a schema change. If a future need genuinely requires a new entity type or field, that is a GL-002 (or GL-006, for `Client Delivery/`) extension, proposed to Silas separately — this file's own §How to extend this Guideline is explicit that neither path runs through here.

**This registry does not cover `Client Delivery/` sources.** Sources that land inside `Client Delivery/` are Warden's, governed by [[GL-006-client-delivery-frontmatter-conventions]] and processed via [[SOP-010-warden-extract-source-to-evidence-pack]] — a structurally similar but separately-scoped job. If a source is client-delivery evidence, this registry does not apply; hand it to Warden.

## Owner / steward model

**Silas owns this registry**, the same schema/frontmatter authority that owns [[GL-002-frontmatter-conventions]] and [[GL-006-client-delivery-frontmatter-conventions]]. This is a deliberate governance choice, not a default: a controlled vocabulary that anyone can edit casually is not a governed vocabulary, it is a folder tree with extra steps — the exact failure Pax's research flags as the difference between taxonomy governance and a category-per-document mess ([[2026-07-11-knowledge-intake-synthesis-hire-research]] §1-2, citing taxonomystrategies.com on governance roles).

- **Silas edits this file.** Adding, retiring, or re-scoping a category, or changing a category's entity-mapping guidance, is Silas's call, proposed to the user via Larry exactly as any GL-002/GL-006 edit is (per Silas's own Critical rule 3 and 8).
- **Cairn is the primary day-to-day consumer, never the editor.** Cairn reads and applies this registry; Cairn does not edit it, mirroring the existing rule that Cairn does not edit GL-002 ([[Team/Cairn - Knowledge Intake Specialist/AGENTS]] §Cross-references, §What Cairn never does).
- **Growth is evidence-gated, not per-item.** A source that does not fit any of the six categories below is not, on its own, grounds to add a seventh. Per Cairn's Method step 6 and Session-Log Discipline, Cairn flags the misfit in its session-log entry (what the source actually was, why none of the six categories fit) rather than inventing an ad-hoc bucket. Silas (via Larry) reviews accumulated flags periodically. **A candidate category is only promoted into this registry once the same genuine misfit has recurred at least twice**, independently, in real intake work — not proposed from a single observed source. This mirrors GL-002 rule 6's own "you will use it more than once" bar for adding a field, and the taxonomy-governance principle of growing a vocabulary only from proven usage, never speculatively ([[2026-07-11-knowledge-intake-synthesis-hire-research]] §1).
- **Retiring a category** follows the same path: Silas edits, documents the reason and the date in §Version history below, never silently drops a row. A retired category does not get its number/name reused for something unrelated, consistent with [[GL-001-file-naming-conventions]] rule 6's non-reuse doctrine for numbered artifacts, applied here in spirit even though categories are named, not numbered.

## The category vocabulary (v1 — six categories)

Six categories, chosen because they are what actually recurs for this user today, checked against the F247 template's own prior attempt for precedent rather than copied from it: the F247 registry's "Source Form" dimension listed a longer, less-governed set (YouTube transcript, meeting transcript, chat export, article/webpage, PDF/report, contract/specification, email/thread, personal note, work note, screenshot, dataset) without ever converging on a fixed, stewarded list. This registry deliberately converges instead — six categories, sized to the 4-7 band professional practice recommends, covering what Cairn's own v1 pilot scope and the two live acquisition adapters (TubeAIR, ICOR course notes) actually feed it, with headroom rather than a category invented per input.

| Category | What it looks like | Typically maps to (GL-002 entity) | Rarely/never |
|---|---|---|---|
| **Article / Written Source** | A blog post, webpage, published essay, or other prose piece authored by someone other than the user, handed to Cairn to extract knowledge from. | **Topic** (the subject the article is about) most often; **Person** or **Organization** when the article is substantively about a specific named individual or company not yet catalogued. | Rarely creates a **Project**, **Goal**, **Habit**, or **Key Element** outright — those are deliberate, user-set entities, not inferred from a single article (see §Note on Goal/Habit/Key Element below). |
| **Document / Report** | A PDF, formal report, whitepaper, or structured reference document. | Two distinct sub-cases, both real: (a) something the user personally owns and must retain (a contract, an ID, an invoice, a warranty, a tax record) → **Document** entity per its `doc_type` enum; (b) an informational or research report to learn from, not to retain as a personal record → treat like an Article, mapping to **Topic** (and possibly **Person**/**Organization** if it names them). | If unsure which sub-case applies, ask: "would the user ever need to physically or digitally *locate* this specific file again as a record?" Yes → Document. No → Topic. |
| **Video / Audio Transcript** | A transcript of a YouTube video, podcast episode, webinar, or recorded talk — the TubeAIR acquisition channel's typical input. | **Topic** (the subject matter) or **Person** (when the transcript is substantively about a specific individual's expertise, persona, or public work) — usually one or the other, rarely both as new creations in the same pass. Worked example: a YouTube transcript of a technical interview usually enriches a Topic or Person note, rarely creates a new entity outright. | Never a **Key Element** directly — Key Elements only come into being via deliberate user-set domains or Topic promotion (see GL-002's Topics schema), never straight from a transcript. |
| **Course / Lesson Note** | A structured learning note tied to a course or curriculum — the ICOR course-note acquisition channel's typical input. | **Topic** (the subject of the lesson) most often; occasionally enriches a **Project** if the course content is directly tied to something the user is actively building. | Never creates a **Key Element** or a **Goal** directly — a course can *feed* a Topic that later gets promoted (GL-002's Topic → Key Element graduation) or *inform* a Goal the user already has, but Cairn does not manufacture either from a lesson note; that is the user's deliberate call. |
| **Chat / Conversation Excerpt** | A pasted chat export or excerpted conversation the user is handing over for filing — general-subject material, not the user's own first-person journaling (that boundary is Penn's, per [[Team/Cairn - Knowledge Intake Specialist/AGENTS]] §Boundary vs Penn). | **Person** (the correspondent) most often; **Project** when the conversation concerns ongoing bounded work; **Topic** when the conversation is exploratory rather than about a specific person or piece of work. | Never `Client Delivery/` Register Items — a conversation excerpt inside a client engagement is Warden's, not Cairn's (see §What this is explicitly NOT above). |
| **Email / Correspondence** | A single email or email thread handed to Cairn directly for filing. | **Person** or **Organization** (the correspondent) most often, following the same logic as Chat/Conversation Excerpt. If the email documents something the user must retain as a record (a booking confirmation, a lease notice), the record itself may separately justify a **Document** entity — file the record as Document, and let the email's substantive content enrich the correspondent's Person/Organization note rather than becoming its own file. | Rarely, on its own, creates a new entity — an email is usually enrichment of an existing correspondent, not a reason to create one. |

## Note on Goal, Habit, and Key Element as destinations

None of the six categories above lists **Goal**, **Habit**, or **Key Element** as a typical primary mapping, and that is deliberate, not an oversight. These three entity types are the user's own deliberately-set operating layer (GL-002 §"The My Life model"), not inference targets from external material. If a source strongly suggests a new Goal, Habit, or Key Element is warranted (a course note that implies a new permanent life domain, an article that crystallizes a measurable target), Cairn does not create it. Per Cairn's Critical rule 7 territory (create-and-enrich, never unilaterally decide the user's operating layer for them) and Cairn's Core philosophy 6, Cairn surfaces the suggestion — in the note body or the session-log entry — for the user to decide, the same restraint already required for entity deletion/merge decisions.

## How this integrates with evidence-origin labeling and backlink-justification

This registry sits at one layer of Cairn's intake pass; it does not replace or restate the other two. All three run on every source, in the order Cairn's own Method already defines:

1. **Classification (this registry)** answers *what kind of source is this, and which entity type does it typically enrich or create* — decided once, stated as the one-line classification per [[Team/Cairn - Knowledge Intake Specialist/AGENTS]] §Method step 2.
2. **Evidence-origin labeling** ([[Team/Cairn - Knowledge Intake Specialist/AGENTS]] §Method step 3, Core philosophy 2) answers a *per-claim* question — how much to trust each individual fact extracted from the source — after this registry has already decided which note the claim is heading into. A category from this registry never upgrades or downgrades a claim's evidence-origin label; the two are independent judgments.
3. **Backlink justification** ([[Team/Cairn - Knowledge Intake Specialist/AGENTS]] §Method step 4, Core philosophy 3) answers a *per-link* question — whether a given `[[wikilink]]` earns its existence — also independent of category. Classifying a source as "usually enriches a Topic or Person" does not grant a free pass to link every named entity the source mentions; each link still needs its own one-line, source-derived reason.

In short: this file tells Cairn *where a source is likely heading*; it does not tell Cairn *what to believe from it* or *what to link to*. Those two disciplines are fully specified in Cairn's own contract and are not restated here.

## Worked example (walking all three layers together)

A pasted YouTube transcript of a technical interview with a named AI researcher arrives.

1. **Classify (this registry):** Video/Audio Transcript. The interview is substantively about the researcher's public work and views — maps to **Person** (the researcher, if not already catalogued) with likely enrichment of an existing or new **Topic** for the subject matter discussed. Stated as the one-line classification in the note.
2. **Evidence-origin label (Cairn's own discipline, not this file):** a direct quote from the researcher in the transcript is labeled as directly present in the material handed to Cairn; a claim Cairn infers about the researcher's broader opinions beyond what was actually said is labeled `Reconstructed / Needs verification`.
3. **Backlink justification (Cairn's own discipline, not this file):** the researcher's employer is mentioned once, in passing, with no further detail — no backlink to that Organization, because there is no stated reason it enriches that note. A named technique the researcher explains at length does justify a backlink to its own Topic note, because the transcript adds real, substantive context there.

## How to extend this Guideline

Three paths, in increasing order of cost — mirroring [[GL-002-frontmatter-conventions]] §"How to extend this Guideline" and [[GL-006-client-delivery-frontmatter-conventions]] §"How to extend this Guideline":

1. **Refine the guidance for an existing category.** Lowest cost. Silas edits the relevant row's "Typically maps to" or "Rarely/never" guidance directly when real usage sharpens the picture. No approval gate beyond Silas's own normal editorial judgment; document the change in §Version history.
2. **Add a new top-level category.** Higher cost, gated by §Owner/steward model above: requires the same genuine misfit to have recurred at least twice in Cairn's session-log flags, approved by the user via Larry, and should keep the total count inside the roughly-4-7 band professional practice recommends. If adding an eighth category would break that band, consider first whether an existing category should split or whether the new case is actually a refinement of §1 rather than a genuinely new category.
3. **This registry is never itself a path to a new `PKM/` entity type or a new frontmatter field.** That is explicitly out of scope here — see §What this is explicitly NOT above. Those changes go through [[GL-002-frontmatter-conventions]] §"How to extend this Guideline" (or [[GL-006-client-delivery-frontmatter-conventions]] for `Client Delivery/`), proposed to Silas, never through this file.
