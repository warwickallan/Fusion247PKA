# TubeAIR changelog

- 2026-07-20 (branch `idea-013/tubeair-transcript-cleanup`) — Feature: transcript readability/cleanup pass. Two deterministic, pure-text functions (no LLM, no network, no paraphrase): `dedupe_rolling_snippets` collapses the auto-caption rolling-window duplication by removing, for each caption line, the leading run of words that is exactly both a suffix of what's already kept and a prefix of the line (word-level exact-overlap removal only — fullest coverage kept, redundant overlaps and exact/contained duplicates dropped; each survivor keeps its own timestamp; idempotent by fixed-point); `reflow_paragraphs` joins the de-duped lines into timestamped paragraphs, breaking on timing gaps (≥ `gap_seconds`, default 2.0s) with a sentence-boundary soft cap so gap-free stretches don't become a wall. §7 of the report now renders a **7.1 Cleaned reading view** ABOVE an unaltered **7.2 Raw captured transcript**; the raw evidence and the immutable GL-011 `Sources (Immutable)/` source are never overwritten (transcript is never replaced by a summary). Added a synthetic 13-line overlapping-caption fixture + 20 tests (dedupe redundancy removal, exact content reconstruction, no invented words, idempotence, no input mutation, reflow gap breaks, timestamp preservation, raw-block-unchanged, immutable-source-untouched). Full suite 44 → 64 tests, all green.

- 2026-07-20 (branch `idea-013/tubeair-telegram-bridge`) — Persistent-service hardening for the inbox + watcher (core `tubeair.py` untouched):
  - **Atomic, corruption-safe state.** Both the inbox idempotency state and the watcher retry ledger now persist via a shared `atomic_write_json` (write `<file>.tmp.<pid>` → fsync → rotate prior good file to `<file>.bak` → `os.replace`, atomic on NTFS/POSIX) — no more torn/partial writes.
  - **Corrupt state fails visibly/safely, never resets to empty.** A shared `load_json_with_backup` recovers from `.bak` on a corrupt/unreadable/wrong-shape file and otherwise raises `StateCorruptError` (loud halt). It never silently returns an empty processed-set (which would re-fetch every already-captured video) or an empty ledger (which would re-open the YouTube-hammering hole). A genuinely fresh run (no primary, no backup) still returns empty.
  - **Per-capture exception isolation.** The inbox loop wraps each capture in try/except and persists state after EACH success, so one raising capture cannot abort the cycle, block later captures, or lose progress already made.
  - **Tests.** Added atomic-write/backup-rotation proofs, corrupt-state and corrupt-ledger fail-loud + recover proofs, per-capture isolation + incremental-persistence proofs, and a starvation proof (a deterministically-raising capture sorted before a healthy one: the healthy one still processes + persists + clears the ledger, while the raiser exhausts at the retry ceiling and stops being fetched). 82 tests pass offline.
  - **CI.** Added `.github/workflows/tubeair-tests.yml` running `test_tubeair` + `test_tubeair_watch` on push/PR touching `tools/tubeair/` (offline monkeypatched path — no API key, no network deps).
- 2026-07-20 (branch `idea-013/tubeair-url-validation-fix`) — Security: URL intake now strictly allowlists YouTube hosts. `parse_video_id` parses with the URL API and checks the parsed `hostname` against an exact-match allowlist (`youtube.com`, `www.youtube.com`, `m.youtube.com`, `music.youtube.com`, `youtu.be`, `youtube-nocookie.com` for `/embed/`), failing CLOSED on lookalike/suffix hosts, userinfo tricks, non-http(s) schemes, and missing hosts; the inbox scanner regex gained a host boundary so lookalike-suffix hosts (e.g. `evilyoutube.com`) no longer match. Added `TestUrlHostAllowlist` covering the hostile and valid cases. Fixes the Codex-flagged bug where non-YouTube hosts were accepted.
